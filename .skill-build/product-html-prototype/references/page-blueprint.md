# Page Blueprint

Use this reference to choose a page shape that matches the product instead of forcing every request into the same layout.

## Layout Patterns

### 1. Operations or admin console

Use when the product helps a team monitor, triage, or coordinate ongoing work.

- top control strip: filters, scope, incident level, date range
- main workspace: queue, map, table, or board
- side rail: details, risk, owner, next step
- lower section: history, audit trail, system explanation, automation rules

This pattern should make urgency, health, and ownership obvious.

### 2. Guided workflow or onboarding page

Use when the product asks a person to complete a sequence.

- progress header: current step, remaining effort, success criteria
- primary stage: form, chooser, or setup action
- helper rail: tips, preview, examples, trust markers
- closing state: confirmation, next step, saved result

This pattern should make progression and confidence obvious.

### 3. Decision or review page

Use when the product exists to compare options or approve something.

- summary header: scope, owner, urgency, target outcome
- comparison surface: cards, split view, timeline, or scenario table
- rationale section: evidence, recommendation, confidence, impact
- decision footer: approve, reject, revise, or delegate

This pattern should make tradeoffs obvious.

### 4. Narrative landing plus interactive preview

Use when the page must sell a concept while also showing how it works.

- narrative hero: promise, audience, payoff
- capability band: three to five big ideas
- embedded prototype surface: mock controls, states, and outputs
- closing proof: KPI targets, rollout phases, or adoption story

This pattern should make the concept memorable.

## State Coverage

Even polished prototypes should hint at system behavior:

- default state
- high-priority state
- blocked or risky state
- empty or first-use state
- confirmation state

Use tabs, toggles, stacked cards, or mini-panels to show these without building multiple pages.

## Content Density Rules

- Dense enterprise tools can support multiple panels if hierarchy is strong.
- Consumer or onboarding flows usually need more whitespace and fewer simultaneous choices.
- Do not add dashboards to every request. Add them only when monitoring is part of the job.

## Visual Direction Rules

- Choose one dominant mood: crisp command center, editorial showcase, calm control room, premium workflow, and similar.
- Use accent colors deliberately; one main accent and one supporting accent is usually enough.
- Give sections distinct rhythm through contrast in scale, spacing, and texture.
- If everything is equally bright, the page will lose its hierarchy.
